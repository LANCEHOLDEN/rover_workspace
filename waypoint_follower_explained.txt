PIDController CLASS (lines 16–45)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  A reusable PID controller used twice: once for heading and once for distance.

  __init__(kp, ki, kd, min_out, max_out)
    kp          — Proportional gain. Reacts directly to current error.
                  Bigger = faster response but can overshoot.
    ki          — Integral gain. Accumulates error over time to eliminate
                  steady-state offset. Bigger = corrects drift but can windup.
    kd          — Derivative gain. Reacts to how fast the error is changing.
                  Bigger = dampens oscillation and overshooting.
    min_out     — Floor on the output (e.g. -2.2 rad/s for angular).
    max_out     — Ceiling on the output (e.g. +2.2 rad/s for angular).
    integral    — Running sum of error × time. Starts at 0.
    prev_error  — Last cycle's error, used to compute the derivative.

  compute(error, dt)
    dt          — Time since last call in seconds. Prevents division by zero.
    integral    — Updated each cycle: integral += error * dt
    Anti-windup — The integral is clamped so it can never grow large enough
                  to saturate the output on its own. Prevents the rover from
                  "winding up" a huge turn command after a long pause.
    derivative  — (error - prev_error) / dt. How quickly the error is changing.
    output      — kp*error + ki*integral + kd*derivative, then clamped to
                  [min_out, max_out].

  reset()
    Clears the integral and prev_error to zero. Called every time a new
    waypoint begins so accumulated state from the previous waypoint doesn't
    carry over.

  Called every time a new /odometry/filtered message arrives (~15–50 Hz).

  FIRST CALL ONLY — Origin Initialization (lines 198–218):
    The very first odometry message sets the local coordinate origin.

    ins_origin      — Stores the raw (x, y) from the EKF. From now on, the rover's
                      position will always be measured relative to this point,
                      so it starts at (0, 0).

    ins_origin_yaw  — Stores the rover's heading at startup.

    Waypoint rotation (lines 207–212):
      Waypoints in the YAML file are written relative to the rover's starting
      heading (e.g. "3 meters straight ahead" = waypoint x=3, y=0).
      If the rover starts facing a different direction, those waypoints would
      be wrong. So all waypoints are rotated by ins_origin_yaw to align them
      with the actual starting heading in the world frame.
      Formula:
        new_x = old_x * cos(yaw) - old_y * sin(yaw)
        new_y = old_x * sin(yaw) + old_y * cos(yaw)

  EVERY CALL — Position Update (lines 220–221):
    current_x = raw_x - origin_x   (offset so rover starts at 0,0)
    current_y = raw_y - origin_y

  EVERY CALL — Yaw Update (lines 223–236):
    Extract raw yaw from the quaternion, then apply EMA smoothing:
      diff = shortest angular distance between new yaw and current smooth yaw
      smooth_yaw += alpha * diff
    The result is stored as current_yaw.

  COMPUTE ERRORS (lines 261–284):
    wp            — The current target waypoint dict.
    dx, dy        — Distance from rover to waypoint in x and y.
    distance      — Straight-line distance to waypoint (Pythagorean theorem).
    target_heading — The angle from the rover TO the waypoint, in radians.
                     atan2(dy, dx) gives the ENU angle: 0=East, π/2=North.
    raw_heading_error — target_heading - current_yaw.
                     Positive = rover needs to turn LEFT (CCW).
                     Negative = rover needs to turn RIGHT (CW).
                     Normalized to [-π, +π].

    Heading error smoothing:
      An EMA is applied to the heading error for fine corrections.
      But for large turns (error > heading_threshold), the raw unsmoothed
      error is used so the rover reacts quickly without lag.

  TIME DELTA (lines 287–289):
    dt = time since last control loop call, in seconds.
    Used by the PID to compute the integral and derivative correctly.

  WAYPOINT REACHED CHECK (lines 292–308):
    If distance < goal_tolerance (0.3 m):
      - Log that the waypoint was reached.
      - Increment current_wp_idx to point to next waypoint.
      - Reset both PIDs (clear accumulated integral/derivative).
      - Reset smooth_heading_error (fresh start for new waypoint).
      - Set turning=True (rover will turn to face new waypoint before driving).
      - Start braking (publish zeros for 0.5 s to actually stop the motors).
      - Reset stuck detector for the next waypoint.
      - Return early (skip velocity command this cycle).

  STUCK DETECTION (lines 310–330):
    Every cycle, check how far the rover has moved since last progress reset.
    If moved >= 0.05 m: reset the stuck timer (we're making progress).
    If NOT moved >= 0.05 m for stuck_timeout seconds:
      Log an error, stop the rover, deactivate.
    This prevents the motors from grinding against an obstacle indefinitely.

  VELOCITY COMMAND GENERATION (lines 332–376):

    cmd = Twist() — Start with zero linear and angular velocity.

    HYSTERESIS (lines 337–342):
      heading_exit_threshold = heading_threshold * 0.4 = ~0.07 rad (~4 deg)
      Enter turn-in-place: when |heading_error| > 0.175 rad (10 deg)
      Exit  turn-in-place: only when |heading_error| < 0.07 rad (4 deg)
      This gap prevents the rover from rapidly switching between turning and
      driving when the error is right near the threshold (jitter/oscillation).

    BRAKE-TO-TURN TRANSITION (lines 346–349):
      When switching from drive mode to turn mode, start braking first.
      Resets the heading PID so there's no stale derivative from driving.

    BRAKING MODE (lines 351–356):
      Counts down brake_cycles_remaining each call.
      Publishes cmd which is all zeros — rover coasts to a stop.
      After 10 cycles (0.5 s), braking ends and turning begins.

    TURNING MODE (lines 357–364):
      angular = heading_pid.compute(heading_error, dt)
        PID output: positive error → positive angular → CCW turn (left).
      Floor: if |angular| > 0.01, boost it to at least min_angular_vel (1.5).
        This ensures small PID outputs aren't too small for the motors to act on.
      linear.x = 0.0 — Pure turn, no forward motion.

    DRIVE MODE (lines 365–374):
      linear = distance_pid.compute(distance, dt)
        PID output: farther away = faster forward speed.
      Floor: speed is at least min_linear_vel (0.2 m/s).
      angular = heading_pid.compute(heading_error, dt)
        Continuously corrects heading while driving forward.
      Reduced angular floor: only half of min_angular_vel (0.75 rad/s) is
        applied during driving, and only when error > 0.15 rad. Avoids
        over-correcting tiny heading wobbles during straight-line driving.

    cmd_pub.publish(cmd) — Send the velocity command to the rover motors.

  1. Startup: Load waypoints from YAML. Wait for first /odometry/filtered.
  2. First odom: Lock in origin (0,0) and rotate waypoints to match heading.
  3. Every 50ms (20 Hz):
       a. Compute distance and heading error to current waypoint.
       b. If distance < 0.3 m → mark waypoint reached, start braking, advance index.
       c. If stuck > 15 s without 5 cm progress → stop.
       d. Decide mode:
            • Braking:  publish zeros for 0.5 s
            • Turning:  PID angular only, no forward motion
            • Driving:  PID forward + PID heading correction
       e. Publish /cmd_vel.
  4. Done: stop rover when all waypoints reached (or loop back to step 3).
